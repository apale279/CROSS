PMApp — Documento di integrazione
1. Scopo dell’applicazione
PMApp è una web app per la gestione sanitaria dei pazienti nei PMA (Posti Medici Avanzati) durante grandi eventi (manifestazioni sportive, concerti, ecc.).

Obiettivi principali:

Veloce, leggera, usabile su tablet/smartphone in campo
Multi-utente e multi-dispositivo sullo stesso paziente senza sovrascritture accidentali
Struttura gerarchica: Manifestazione → PMA → Paziente
Non espone un backend REST proprio: tutta la logica dati passa da Firebase (Auth + Firestore) lato client.

3. Gerarchia dati (modello mentale)
Manifestazione (evento)
  ├── stato: APERTA | CHIUSA (se CHIUSA → dati in sola lettura)
  ├── impostazioni (liste cliniche, preset, consensi, partecipanti…)
  ├── contatori/pazienti_progressivo → genera ID visibili P_1, P_2…
  └── PMA (1 o più per manifestazione)
        ├── impostazioni_pma (posti_letto, farmaci_usati)
        └── Pazienti (UUID Firestore, ID visibile P_n)
Tenant / scope:

id_manifestazione → isola tutto ciò che appartiene a un evento
id_pma → isola pazienti e alert operativi
Utenti legati a manifestazione e/o PMA tramite profilo Firestore
4. Autenticazione e profilo utente
Firebase Auth
Login email/password
Persistenza: browserSessionPersistence (logout implicito alla chiusura tab)
Superadmin: timeout idle 10 minuti
Documento profilo: utenti/{uid}
{

  email: string            // da Auth
  nome: string
  rank: UserRank
  id_manifestazione?: string  // obbligatorio per Centrale e staff operativo

Regola architetturale: profilo utente letto con getDoc, mai onSnapshot.

7. Regole Firestore (security)
Attualmente permissive: qualsiasi utente autenticato può leggere/scrivere su tutte le collezioni principali.
Eccezione parziale: codici_minori richiede rank valido in utenti/{uid}.rank.

Isolamento tenant: enforced lato app (query filtrate su id_manifestazione / id_pma), non nelle rules.

8. Pattern di sincronizzazione (CRITICO per integrazione)
onSnapshot (real-time)  →  pazienti, dashboard PMA, dashboard manifestazione, allerte, codici_minori, rubrica, file utili
getDoc / getDocs          →  utenti, impostazioni (caricamento iniziale), manifestazioni (liste)
updateDoc parziale        →  SEMPRE per salvataggi paziente (mai overwrite intero documento)
runTransaction            →  creazione paziente + incremento progressivo P_n
Motivazione: più operatori medici possono lavorare contemporaneamente sulla stessa scheda.

12. Punti di integrazione con un'altra app
A. Condivisione Firebase (approccio più semplice)
Se l'altra app usa lo stesso progetto Firebase:

Leggere/scrivere le stesse collezioni rispettando lo schema sopra
Usare updateDoc parziale su pazienti
Rispettare id_manifestazione e id_pma come chiavi di scope
Non usare setDoc full-document su pazienti esistenti
B. Progetti Firebase separati (sync)
Serve un bridge (Cloud Functions, cron, webhook) che:

Mappa entità tra i due sistemi (tabella external_id o campo custom)
Replica eventi: nuovo paziente, cambio stato, dimissione, invio PS
Gestisce conflitti con timestamp/updated_at (attualmente non c'è un campo updated_at globale sul paziente — andrebbe aggiunto se serve sync bidirezionale)
C. Campi utili da aggiungere per integrazione cross-app
Se l'altra app deve collegarsi, considerare campi opzionali su pazienti:

external_app_id?: string      // ID nell'altra app
external_sync_at?: Timestamp
external_source?: string      // nome app sorgente
D. Eventi chiave da propagare
Creazione paziente (P_n assegnato)
Cambio stato / codice_colore
Dimissione (dimissione_esito, dimesso_at)
Invio PS (invio_ps_*)
Chiusura scheda (aperto: false)
E. Auth condivisa
Stesso Firebase Auth UID → stesso utenti/{uid}
Oppure custom claims / mapping UID tra progetti
F. File e media
ECG e file utili: URL Cloudinary pubblici (secure_url)
Firme: base64 inline nel documento Firestore (non URL esterni)
13. Indici Firestore configurati
pma:           id_manifestazione ASC
pazienti:      id_pma ASC
codici_minori: id_pma ASC + ora_accesso DESC
Query comuni:

Pazienti per PMA: where('id_pma', '==', pmaId)
PMA per manifestazione: where('id_manifestazione', '==', manId)
Pazienti dimessi: where('id_manifestazione', '==', manId) + filtro client-side stato == 'dimesso'
14. Vincoli e note per l'integratore
Nessuna REST API: integrazione via Firestore SDK, Admin SDK, o Cloud Functions
Naming convention: snake_case su Firestore, ID manifestazione senza spazi
Manifestazione CHIUSA = read-only in UI (rispettare in integrazioni)
Multi-tenancy soft: sempre filtrare per id_manifestazione / id_pma
Progressivo paziente (P_n) è per-manifestazione, non globale
Rank matrix complessa: permessi tab scheda variano per ruolo (vedi src/lib/rankMatrix.ts)
Offline: Firestore con cache persistente IndexedDB + multi-tab
Repo: PMApp — SPA deployabile su Firebase Hosting / Vercel / qualsiasi static host
