# Contesto: applicazione CROSS
## Cos’è CROSS
CROSS è una **centrale operativa web** (React + Vite + Firebase) per coordinare **soccorso sanitario** durante manifestazioni ed eventi (gare, concerti, fiere, ecc.).
Funzioni principali:
- Gestione **eventi** (interventi sanitari) con priorità colore (Bianco/Verde/Giallo/Rosso)
- Assegnazione **missioni** ai **mezzi** (ambulanze, auto mediche, moto mediche)
- Anagrafica e stato **pazienti** (esiti, trasporto, valutazioni cliniche MSB/MSA)
- **Dashboard** operativa in tempo reale (tabelle eventi/missioni, stato mezzi, mappa Google)
- **Mappa tattica** su piantina (Leaflet, coordinate % su immagine)
- **Diario** operativo con note importanti in ticker
- **Bot Telegram** per equipaggi: ricezione missioni, avanzamento stati, GPS, SOS
- **Kiosk** (pagine `/kiosk/*`) per monitor esterni in sola lettura
- **Impostazioni** per tenant: tipi evento, ospedali, stazionamenti, registro partecipanti, Telegram, PDF guida
Stack: React 19, React Router 7, Tailwind 4, Firebase (Auth + Firestore + client SDK), Firebase Admin nelle API Vercel (`/api/*`), Google Maps, Leaflet, deploy tipico su **Vercel**.
---
## Modello multi-tenant
Ogni installazione può avere una o più **manifestazioni** (tenant). Tutti i dati operativi vivono sotto:
`manifestazioni/{tenantId}/...`
- **tenantId** = ID documento Firestore in collezione root `manifestazioni`
- In produzione si imposta `VITE_TENANT_ID`; se omesso e esiste un solo documento in `manifestazioni`, viene scelto automaticamente (`TenantContext`)
- Ogni documento manifestazione ha metadati: `nome`, `dataInizio` (Timestamp), `luogo`, `creatoIl`
**Non confondere:**
- `idEvento` / `idMissione` / `idPaziente` = ID progressivi leggibili (E1, M2, P3)
- `idUnivoco` = UUID stabile per collegamenti tra entità (preferire questo per integrazioni)
- `_docId` = ID documento Firestore (auto-generato, usato dall’app per update/delete)

| `equipaggio` | string | Snapshot testuale equipaggio al creare missione |

| `tratteMissione` | array | `{ id, descrizione, quando }` tappe operative |
| `codiceColore` | string? | Da valutazione MSB paziente |
| `missioneEccezioneMotivo` | string? | `DIROTTAMENTO`, `FLAG_DOWN`, `AVARIA_SINISTRO` |
**Glossario stati missione (ordine tipico):**
`ALLERTARE` → `ALLERTATO` → `PARTITO` → `IN POSTO` → `DIRETTO H` → `ARRIVATO H` → `RIENTRO` → `FINE MISSIONE`
Eccezione: `ANNULLATA` (dirottamento, flag-down, avaria)
**Side effects al cambio stato:**
- `FINE MISSIONE` → mezzo `statoMezzo: "Disponibile"`
- `ANNULLATA` + motivo avaria → mezzo `"Non operativo (avaria/sinistro)"`, `operativo: false`
- `ARRIVATO H` → tutti i pazienti dello stesso evento/mezzo con esito `"Trasporta"` passano a stato `ARRIVATO H`
- Creazione missione → mezzo `statoMezzo: "Non disponibile"`
- Cambio stato da centrale → notifica Telegram equipaggio (API server)

## Relazioni tra entità (per integrazioni)
Evento (idUnivoco) ├── Missioni (eventoIdUnivoco + eventoCorrelato=idEvento) │ └── Mezzo (sigla = missione.mezzo) └── Pazienti (eventoIdUnivoco + eventoCorrelato)

Paziente in trasporto → missioneIdUnivoco / idMissione + mezzo Missione ARRIVATO H → sync pazienti Trasporta stesso evento+mezzo

**Query consigliate per integrazione:**
- Filtrare per `manifestationId` (sempre presente sui documenti)
- Collegare preferendo `idUnivoco` (evento/missione/paziente)
- Per mezzi usare **sigla** come chiave primaria
- Ascoltare in real-time le sottocollezioni sotto `manifestazioni/{tenantId}/`

## API server (Vercel, richiedono Firebase ID token Bearer + tenantId nel body)
Endpoint principali in `/api/`:
- `telegram-webhook` — bot Telegram (stati, GPS, SOS, auth)
- `telegram-send` — invio manuale missione a equipaggio
- `telegram-notify-stato` — notifica cambio stato
- `telegram-set-password`, `telegram-force-logout`, `telegram-wipe`, `telegram-logged-users`
- `telegram-broadcast-nota` — diario → tutti equipaggi loggati
- `active-users` — sessioni web attive
- `clear-mezzi-gps` — pulizia posizioni GPS
- `piantina-upload`, `guida-pdf-upload` — asset su storage esterno
Il tenant nelle API si risolve da body (`tenantId` / `manifestationId`) o env `TELEGRAM_TENANT_ID` / `VITE_TENANT_ID`.

## Autenticazione
- **Web:** Firebase Auth (email/password). Nessuna auto-registrazione.
- **Firestore rules attuali:** `allow read, write: if true` (aperte — in produzione andrebbero restringate).
- **Telegram:** password condivisa per manifestazione (hash su impostazioni) + associazione chatId ↔ mezzo.

## Linee guida per integrare un’altra app
1. **Usare sempre il path tenant:** `manifestazioni/{tenantId}/...`
2. **Non scrivere sulle collezioni root legacy.**
3. **Rispettare i side effect** (stato mezzo, sync pazienti ARRIVATO H, auto-chiusura evento) o replicarli esplicitamente.
4. **Identificatori:** esporre/consumare `idUnivoco` tra sistemi; `idEvento`/`M`/`P` sono solo display.
5. **Real-time:** Firestore `onSnapshot` sulle collezioni tenant (come `ManifestazioneDataContext`).
6. **Posizioni:** non mescolare `stazionamento`, `coordinate_piantina`, `posizioneReale`.
7. **Telegram:** integrazioni esterne che modificano stati missione dovrebbero allinearsi al flusso stati e opzionalmente chiamare le API notify o aggiornare Firestore in modo coerente con `buildStatoChangeFields` / `storicoStati`.
8. **Capacità mezzo:** se l’altra app assegna trasporti, rispettare il limite di 3 pazienti «Trasporta» per mezzo.

## Dominio in una frase
CROSS modella un’**emergenza sanitaria** come **Evento** → una o più **Missioni** (mezzi inviati) → **Pazienti** valutati/trasportati, con **stati missione** che guidano logistica e notifiche campo, e **configurazione per manifestazione** in un unico tenant Firestore isolato.
